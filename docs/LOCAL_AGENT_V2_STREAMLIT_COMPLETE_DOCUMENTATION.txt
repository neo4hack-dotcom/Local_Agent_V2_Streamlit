====================================================================
1) SYSTEM OVERVIEW
====================================================================
Local Agent Studio is a fullstack local AI orchestration platform built around
LangGraph and FastAPI, with a React frontend. It lets users:
- Configure LLM providers (Ollama or generic HTTP endpoint).
- Configure multiple databases (ClickHouse, Oracle, Elasticsearch).
- Create and manage multiple specialized agents.
- Execute either:
  - Single-agent runs.
  - Multi-agent manager orchestration with real-time event streaming.
- Optionally forward orchestration events to an external webhook UI.
- Optionally export manager intermediate results to Excel.
- Define automation rules that trigger agent chains on file events.

Core architectural principle:
- Manager plans and coordinates.
- Specialized agents execute bounded tasks.
- Every important orchestration action emits explicit events.

4) BOOTSTRAP, STARTUP, SHUTDOWN
====================================================================
Backend startup sequence (backend/app/main.py):
1. Build default settings and default agents if storage files are missing.
2. Instantiate JSON repositories:
   - settings: backend/data/settings.json
   - agents: backend/data/agents.json
3. Instantiate AutomationEngine with settings+agents repos.
4. Configure CORS (FRONTEND_ORIGIN env, default http://localhost:5173).
5. Load repositories at startup event.
6. Start automation engine background thread.
7. Register routers:
   - /api/config
   - /api/databases
   - /api/agents
   - /api/automations

Shutdown sequence:
1. Stop automation engine thread.

Health endpoints:
- GET /health (backend)
- GET /health (bridge app)

10) LANGGRAPH WORKFLOWS
====================================================================
10.1 SQL Analyst LangGraph
File: backend/app/core/langgraph_agent.py

State keys:
- question, agent, database, llm, schema, sql, rows, answer.

Node sequence:
1. schema:
   connector.schema_snapshot(allowed_tables)
2. sql:
   LLM prompt from sql_prompt_template + schema + allowed tables.
   SQL extracted from markdown fences if needed.
3. query:
   connector.run_query(sql, limit=max_rows)
4. answer:
   LLM prompt from answer template with JSON rows payload.

Graph edges:
START -> schema -> sql -> query -> answer -> END

Serialization detail:
- datetime/date/time in rows are converted through custom JSON default.

10.2 Web Navigator LangGraph
File: backend/app/core/web_navigation_agent.py

State keys:
- task, start_url, url, page_content, history, status, error,
  step, max_steps, allowed_domains, next_action, final_answer,
  final_rows, browser session.

Node sequence:
1. bootstrap:
   open start URL.
2. read_page:
   collect simplified page snapshot.
3. plan_action:
   LLM outputs strict JSON action:
   navigate|click|fill|extract|finish.
4. execute_action:
   execute browser action through Playwright session.
5. finalize:
   LLM writes concise final report from history + snapshot.

Routing rules:
- After plan: finalize when status terminal or action=finish.
- After execute: loop to read_page unless terminal.

Runtime prerequisites:
- Playwright package + Chromium binaries installed.

11) DATABASE CONNECTORS AND ROUTING
====================================================================
11.1 Routing policy
Function: resolve_database_for_agent(...)
Resolution order:
1. requested database_id
2. agent template_config.database_id
3. agent template_config.database_name
4. active database (if compatible)
5. first compatible profile by engine

Compatibility constraints:
- sql_analyst -> clickhouse or oracle
- clickhouse_table_manager -> clickhouse only

11.2 Connector behavior
ClickHouseConnector:
- test_connection: SELECT 1
- schema snapshot from system.columns
- run_query enforces read-only SQL + auto LIMIT if absent
- run_statement supports SELECT and command execution

OracleConnector:
- requires username/password + dsn or host/port/database
- test_connection via dual
- schema snapshot from all_tab_columns
- run_query wraps SQL with ROWNUM <= limit
- run_statement unsupported in table-manager mode

ElasticsearchConnector:
- HTTP cluster health check
- run_query/run_statement not supported for SQL agents
- schema_snapshot returns informational message only

11.3 Read-only guardrail
- SQL analyst queries must start with SELECT.
- Write keywords are blocked by regex (insert/update/delete/drop/alter/etc.).

12) AGENT EXECUTOR ENGINE (ALL AGENT TYPES)
====================================================================
Dispatcher class: AgentExecutor
Core entrypoint: execute(agent, question, database)

12.1 SQL Analyst
Modes:
- llm_sql (default): executes LangGraph SQL pipeline.
- parameterized_template: fills placeholders in SQL template.

Parameterized mode pipeline:
1. Normalize parameter schema from template config.
2. Extract parameter values via LLM JSON + regex fallback.
3. Coerce values by declared type (string/integer/number/boolean/date).
4. Render SQL literals safely.
5. Execute query.
6. Ask LLM to produce business answer from SQL + rows.

Failure conditions:
- Missing required parameters.
- Type coercion failure.
- Unsupported DB engine.

12.2 ClickHouse Table Manager
Pipeline:
1. Build safety policy from template config.
2. Read schema snapshot.
3. Ask LLM for strict JSON SQL operations.
4. Parse operations (JSON, fenced SQL, or split fallback).
5. Validate each statement against policy.
6. Execute statements with optional stop_on_error.
7. Produce operational summary via LLM.

Policy constraints:
- Always-blocked operation classes: grant/revoke/attach/detach/system/kill.
- Optional protections for destructive schema operations.
- Fine-grained insert/update/delete allow flags.
- max_statements cap.

12.3 Unstructured to Structured
- Uses output_schema in template_config.
- Can enforce strict JSON response.
- Returns parsed JSON or fallback with raw output.

12.4 Email Cleaner
- Cleans noisy text to structured sections.
- Controlled by max bullets + include_sections.

12.5 File Assistant and RAG Context
Common engine:
- Read files from folder_path.
- Filter by extensions/size/count.
- Score by token overlap.
- For RAG: chunk text and rank chunks.
- Build context snippets and ask LLM for grounded answer.

12.6 Text File Manager
Operations:
- list, read, create, write, append.

Security constraints:
- All paths are forced inside configured root (safe path resolver).
- Optional auto-create folder.
- Optional overwrite policy.

12.7 Excel Manager
Operations:
- create_workbook, list_sheets, read_sheet, append_rows, set_cells.

Execution details:
- Uses openpyxl.
- Can auto-create workbook.
- Validates rows and cell updates.
- Repairs low-quality LLM actions when possible.

12.8 Word Manager
Operations:
- create_document, read_document, append_paragraphs, replace_text, list_documents.

Execution details:
- Uses python-docx.
- Can auto-create document.
- Generates concrete prose when action payload is placeholder-like.

12.9 Elasticsearch Retriever
- Uses _search endpoint with simple_query_string.
- Supports credentials/api key from template config or DB profile.
- Returns ranked hits + LLM-grounded summary.

12.10 Internet Search
- DuckDuckGo HTML search + optional instant API fallback.
- Returns top_k rows with title/url/snippet.

12.11 RSS News Agent
Pipeline:
1. Fetch RSS/Atom feeds.
2. Parse items and publication dates.
3. Filter by lookback, interests, exclude keywords.
4. Score by interest match + recency.
5. Deduplicate and select top_k.
6. Produce briefing via LLM.

12.12 Web Scraper
Pipeline:
1. Build candidate URLs from config + question URLs/domains.
2. Optional search fallback.
3. Crawl pages (optional link following).
4. Extract title/text snippets.
5. Produce answer from scraped evidence.

12.13 Web Navigator
- Delegates to WebNavigationRunner LangGraph browser workflow.

12.14 Wikipedia Retriever
- Query MediaWiki API, with fallback to OpenSearch on rate/forbidden scenarios.
- Retrieves summaries and links, then asks LLM for cited synthesis.

13) MULTI-AGENT MANAGER SPECIFICATION
====================================================================
Component: MultiAgentManager

13.1 Runtime state
- enabled_agents: currently enabled catalog subset.
- unavailable_agents: runtime-marked unavailable agent map.
- history: execution observations used for planning and fallback.
- conversation_memory: normalized prior turns.

13.2 Event model
The manager streams these events:
- manager_start
- manager_decision
- manager_warning
- agent_call_started
- agent_call_completed
- agent_call_failed
- agent_marked_unavailable
- manager_final

13.3 Decision loop algorithm
Step-by-step:
1. Emit manager_start with budget and available agents.
2. If no plannable agents, finalize as blocked.
3. For each step until limits:
   a. Ask LLM for strict JSON decision (continue/done/blocked + calls).
   b. Deduplicate calls by signature.
   c. Adapt decision for progressive data analysis if needed.
   d. Emit manager_decision.
   e. If done/blocked -> emit manager_final.
   f. Execute planned calls sequentially (budget-guarded).
   g. Emit started/completed/failed events.
   h. Optionally mark agent unavailable after runtime failure patterns.
   i. Try early finalization gate after successful results.
4. If limits exhausted, finalize with status exhausted.

13.4 Data-analysis adaptation layer
Purpose:
- Avoid premature finalization in SQL analytical requests.

State tracking includes:
- successful SQL calls count
- tables touched
- columns observed
- where-clause coverage
- cross-table query count

Behavior:
- If evidence is incomplete, manager injects SQL bootstrap calls.
- Encourages progression:
  table discovery -> scoped filtering -> cross-table validation -> synthesis.

13.5 Finalization and fallback behavior
- Normalizes final answer to avoid placeholder values (none/null/empty).
- If missing, uses latest reliable successful answer.
- Generates manager_summary (LLM) with strict structure fallback.
- Generates sanity judge report (LLM) with heuristic fallback.

13.6 Missing-information inference
Priority:
1. Runtime unavailable agents summary.
2. Recent blocking errors.
3. Generic capability/budget shortfall suggestion.

16) AUTOMATION ENGINE PROCESS
====================================================================
Component: AutomationEngine

16.1 Thread lifecycle
- start(): starts daemon loop thread.
- stop(): stops loop via event and join timeout.

16.2 Rule scanning loop
For each enabled automation rule:
1. Check poll interval gate.
2. Snapshot watched files (recursive or not).
3. Detect newly seen paths compared to previous snapshot.
4. Limit processed events per scan.
5. For each new file, execute configured agent chain.

16.3 Agent chain execution
For each agent id in rule.agent_chain:
1. Resolve enabled agent.
2. Resolve compatible database (if required).
3. Execute agent with current prompt/question.
4. Store step result.
5. If successful answer exists, feed it as next question to subsequent agent.
6. On failure, stop chain and log partial/failed status.

16.4 Automation run logs
Stored in memory deque with:
- metadata (rule id/name, file path, timestamps)
- per-step statuses
- final answer and error summary

16.5 Error-throttling behavior
Repeated watch-path errors are rate-limited by cooldown to avoid log spam.

2. Frontend deps installed.
3. Backend venv created and dependencies installed.
4. Playwright Chromium installation attempted.

19.2 Basic configuration process
1. Open Settings tab.
2. Configure LLM and validate model connection.
3. Create at least one database profile.
4. Set active database.
5. (Optional) configure webhook and test it.

19.3 Agent design process
1. Open Agents tab.
2. Create new agent from template.
3. Fill minimal operational fields.
4. Save and optionally refine advanced JSON/prompt sections.
5. Verify agent appears enabled.

19.4 Single-agent execution process
1. Open Playground in single mode.
2. Select agent and enter question.
3. Submit request.
4. Backend routes through AgentExecutor.
5. UI displays answer, SQL (if any), rows, details.

19.5 Manager execution process
1. Open Playground in manager mode.
2. Enter request (optional memory + Excel export flag).
3. Stream starts and timeline updates in real-time.
4. Manager orchestrates calls and emits events.
5. Final event includes answer, summary, judge outputs.
6. If enabled, Excel path/error is shown.

19.6 Automation execution process
1. Configure rule with watch path and agent chain.
2. Engine scans for new files at poll interval.
3. New file event triggers chained agent execution.
4. Run logs visible in configuration tab.

20) OPERATIONS, OBSERVABILITY, TROUBLESHOOTING

- LLM test: /api/config/llm/test
- DB test: /api/databases/{id}/test
- Network probe: /api/config/network/test

20.2 Typical failures and causes
- “Object of type datetime is not JSON serializable”:
  solved by safe serialization helpers in manager/export/SQL paths.
- Web navigator unavailable:
  Playwright package or Chromium missing.
- Manager has summary but no answer:
  final-answer normalization and fallback logic should now backfill from best
  successful observation or status-default string.
- SQL agent blocked:
  non-SELECT query or unsupported engine profile.

20.3 Safe runtime practices
- Keep destructive operations disabled by default in ClickHouse manager.
- Use DB profile scoping via template_config.database_id.
- Keep webhook disabled in environments without secure outbound control.

21) EXTENDING THE APPLICATION (ENGINEERING PROCESS)
====================================================================
To add a new agent type safely, follow this sequence:
1. Add backend AgentType literal in models.py.
2. Add default template in agent_templates.py.
3. Implement execution method in AgentExecutor and dispatch branch.
4. Add required DB routing rule if applicable.
5. Update frontend AgentType union and assisted UI fields in AgentsTab.
6. Validate end-to-end from single mode and manager mode.
7. Add/update docs and import/export compatibility.

To add a new manager event:
1. Emit event through manager `_event` helper.
2. Ensure front timeline cards can parse and render it.
3. Ensure webhook consumer compatibility.
4. Update type unions and docs.

To add a new connector engine:
1. Extend DatabaseProfile engine literal.
2. Add connector implementation + connector_for mapping.
3. Update database routing compatibility logic.
4. Add UI engine option and draft defaults.

23) STEP-BY-STEP CHAINING MAPS (FUNCTIONAL)
====================================================================
23.1 Single agent run chain
UI `PlaygroundTab.executeSingle` -> API `runAgent` -> backend route `/agents/{id}/run`
-> database resolution -> `AgentExecutor.execute` -> typed handler -> return response
-> UI updates conversation memory and renders details.

23.2 Manager stream chain
UI `PlaygroundTab.executeManager` -> API `runManagerStream`
-> backend route `/agents/manager/run/stream`
-> `MultiAgentManager.run_stream` event generator
-> each NDJSON line parsed by frontend and appended to timeline
-> final manager event stored in `managerFinal` with summary/judge fields.

23.3 Automation chain
AutomationEngine loop -> detect new file -> build prompt
-> sequential `AgentExecutor.execute` for each chain item
-> step logs + final run status stored in deque
-> UI reads via `/automations/{id}/runs`.

23.4 Webhook chain
Manager event produced -> router forwards through WebhookDispatcher
-> envelope added (run context + sequence + optional timeline)
-> external system receives payload.

23.5 Open WebUI bridge chain
External webhook sends manager event to bridge
-> bridge transforms into chat text payload
-> forwards to Open WebUI endpoint.

APPENDIX F - DETAILED FUNCTION BEHAVIOR NOTES (STEP-BY-STEP)
====================================================================
This appendix complements the raw inventory by adding practical behavior notes.
The intent is operational clarity: what each important function does, in what
order, with what constraints.

F.1 backend/app/main.py
- _default_settings:
  returns default AppSettings object used when settings file is missing.
- _default_agents:
  seeds the repository with one SQL analyst default agent.
- startup:
  loads settings/agents repositories and starts AutomationEngine thread.
- shutdown:
  stops AutomationEngine.
- health:
  backend liveness probe.

F.2 backend/app/api/deps.py
- get_settings_repo/get_agents_repo/get_automation_engine:
  FastAPI dependency providers returning app state singletons.

F.3 backend/app/api/routers/config.py
- get_config/get_llm_config/get_manager_config/get_webhook_config:
  retrieve current persisted settings sections.
- update_llm_config/update_manager_config/update_webhook_config:
  mutate corresponding section and persist via repository save.
- test_webhook_delivery:
  validates configured webhook by sending synthetic event.
- test_llm_connection:
  validates provider reachability and model listing logic.
- list_llm_models:
  returns selectable model names for UI dropdowns.
- test_network_access:
  backend outbound connectivity diagnostics (DNS + HTTP probes).

F.4 backend/app/api/routers/databases.py
- _find_database:
  strict lookup helper, raises 404 when missing.
- _validate_unique_database_ids:
  import payload integrity check.
- create_database:
  allocates UUID id and optionally sets first profile as active.
- update_database:
  full replacement update by id.
- delete_database:
  removes profile and reconciles active_database_id fallback.
- set_active_database:
  validates target exists and updates active pointer.
- test_database_connection:
  calls connector_for(profile).test_connection().
- export_databases_config:
  emits versioned payload with active ID and all profiles.
- import_databases_config:
  replace or merge mode with deterministic result counts.

F.5 backend/app/api/routers/automations.py
- _find_automation:
  strict lookup helper.
- create/update/delete automation:
  persist rule changes and reset engine per-rule runtime cache.
- list_automation_runs:
  validates rule existence then returns bounded in-memory logs.

F.6 backend/app/api/routers/agents.py
Shared utility functions:
- _find_agent/_find_agent_index:
  strict lookup helpers.
- _append_agent_audit:
  prepend new snapshot and enforce max 5 stored versions.
- _normalized_conversation_history:
  latest-turn and per-turn character bounding.
- _contextual_question:
  prepends memory transcript to current question with instruction text.
- Excel export helpers:
  _safe_json_text, _excel_cell_value, _call_event_key,
  _export_manager_intermediate_results_to_excel.

Endpoint execution functions:
- list_agents/get_agent_templates:
  read-only listing endpoints.
- export_agents_config/import_agents_config:
  config transfer with replace/merge and optional audit preservation.
- create_agent/update_agent/delete_agent:
  CRUD with update audit tracking.
- list_agent_audit/restore_agent_version:
  version timeline and rollback mechanics.
- run_agent:
  resolves database + memory, executes AgentExecutor, wraps response.
- run_with_manager:
  sync manager run, optional Excel export, optional webhook fan-out.
- run_with_manager_stream:
  NDJSON streaming with fallback manager_final on unexpected errors.

====================================================================
APPENDIX G - CONSTRAINT MATRIX (TECHNICAL AND OPERATIONAL)
====================================================================
G.1 API contract constraints
- ManagerRunRequest max_steps: [1..30]
- ManagerRunRequest max_agent_calls: [1..100]
- ConversationTurn content length: [1..6000]
- LLM timeout_seconds: [1..300]
- LLM temperature: [0.0..2.0]
- Webhook timeout_seconds: [1..60]

G.2 Storage and safety constraints
- JSONRepository writes are atomic (tmp+replace).
- Agent audit snapshots retained: max 5 versions per agent.
- Automation logs retained in memory: max 400 runs.
- Manager final webhook timeline cap: 400 events.

G.3 Execution constraints
- SQL analyst: SELECT-only enforcement.
- ClickHouse manager: policy-based DDL/DML control.
- Web navigator: requires Playwright runtime.
- Excel export: requires openpyxl dependency.
- Word manager: requires python-docx dependency.

G.4 UI constraints and defaults
- Conversation memory capped at 12 turns in Playground.
- Manager stream must end with manager_final event.
- External webhook replacement blocks local Playground execution by design.

G.5 Operational constraints
- FRONTEND_ORIGIN must match frontend host for CORS.
- OPEN_WEBUI_WEBHOOK_URL required for bridge forwarding.
- Database routing can fail when agent type and DB engine are incompatible.
